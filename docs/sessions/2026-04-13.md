### 2. Decisiones de arquitectura tomadas — vendedores desde Odoo

**Decisión:** los vendedores se crean en Odoo y el gateway los importa automáticamente. Odoo es la fuente de verdad de los empleados.

**Por qué:** el equipo de administración ya vive en Odoo. No tiene sentido gestionar los 120 vendedores en dos lugares. Un solo punto de entrada.

**Modelo elegido: `nexus.vendor` (propuesta de Lenn)**
En lugar de extender `hr.employee`, Lenn creó un modelo dedicado en Odoo. Ventajas:
- Sin dependencia del módulo RRHH
- Campos exactamente los que NEXUS necesita
- `nexus_uuid` autogenerado en Odoo = `vendedores.uuid` en PostgreSQL
- Limpio y aislado

**Vinculo entre sistemas:**
```
Odoo: nexus.vendor.nexus_uuid  ←→  PostgreSQL: vendedores.uuid
Odoo: nexus.vendor.id          ←→  PostgreSQL: vendedores.odoo_vendor_id
```

### 3. Clarificaciones de arquitectura importantes

**Dirección de la comunicación — solo un sentido:**
```
Gateway → Odoo   (siempre, el gateway inicia)
Odoo    → Gateway  (nunca — Odoo no sabe que el gateway existe)
```
Odoo es pasivo. El gateway jala data con crons y empuja pedidos/pagos/visitas con BullMQ.

**Protocolo: JSON-RPC puro**
```
POST /web/session/authenticate  ← login del bot
POST /web/dataset/call_kw       ← todas las operaciones ORM
```
El mismo protocolo que usa el browser de Odoo internamente. Funciona con 1 sola cuenta, sin configuración especial en Odoo.

**Por qué NO usamos webhooks desde Odoo:**
Agregaría complejidad en el lado de Odoo, requeriría módulos extra y mantenimiento adicional. Los crons del gateway son suficientes para este caso de uso y mantienen Odoo sin tocar.

**El mapa Leaflet del módulo nexus_field:**
El JS corre en el browser del supervisor — no es Odoo el que llama al gateway, es el navegador del usuario. Odoo server no participa.

### 5. Cron sync vendedores desde Odoo — implementado

Archivo: `crons/sync_vendors.js`

**Frecuencia:** cada 1 hora + al arrancar el servidor

**Lógica:**
```
1. Lee nexus.vendor en Odoo donde active = true
2. Por cada vendedor:
   └── Nuevo    → INSERT vendedores + password temporal en logs
   └── Existente→ UPDATE si cambió nombre, email o zona
   └── Removido → activo = false (JWT inválido en próximo request)
3. Si nexus.vendor no existe en Odoo → error claro en logs
```

**Campos que lee de Odoo:**
```javascript
{ fields: ['id', 'nexus_uuid', 'name', 'email', 'zone'], limit: 500 }
```

El `email` viene de `partner_id.email` como campo related stored en Odoo — el cron lo lee sin join extra.

## Decisiones técnicas relevantes del día

### Por qué `nexus.vendor` y no `hr.employee`
`hr.employee` requiere el módulo RRHH de Odoo instalado y tiene campos que no son relevantes para NEXUS. `nexus.vendor` es un modelo propio en el módulo `nexus_field` con exactamente los campos que el gateway necesita: nombre, email (related de partner), zona, nexus_uuid. Cero dependencias externas.

### Por qué el UUID lo genera Odoo y no el gateway
El vendedor existe primero en Odoo. Odoo autogenera `nexus_uuid` al crear el `nexus.vendor`. Ese UUID pasa al gateway en el primer sync y se convierte en `vendedores.uuid`. El gateway nunca necesita generar un UUID para un vendedor — lo recibe de Odoo y lo preserva. Garantiza que el mismo vendedor tiene el mismo UUID en ambos sistemas.

### Por qué no hay fallback a `hr.employee` en el cron
Si el módulo no está instalado el error debe ser visible y claro. Un fallback silencioso a todos los empleados podría crear vendedores fantasma en producción. Mejor fallar ruidosamente que tener datos incorrectos.
