## Decisiones de arquitectura tomadas hoy

### GPS sin internet
- GPS = satélite puro, funciona sin datos móviles
- Flutter almacena coordenadas de clientes en Drift SQLite durante sync inicial
- Geofence calculado ON DEVICE — sin red
- Check-in automático al entrar en radio de 100m del cliente
- Outbox P0 envía trail de ubicaciones al detectar cualquier señal
- Redis guarda `location:{vendedor_id}` con TTL 2h

### Módulo Odoo no toca la DB de Odoo para ubicaciones
```
Browser en Odoo → JS hace fetch al Gateway → Redis → respuesta <10ms
Odoo DB = 0 queries de ubicación
```

### Paradas flexibles
- `orden` = plan original del cron 4AM (inmutable)
- `orden_actual` = orden real según avance del vendedor
- Estados: `pending → on_site → completed` / `pending → skipped → pending`

### Conflicto de precios post-Odoo-caído (PENDIENTE de implementar)
- Si diferencia de precio < 5% → enviar a Odoo igual
- Si diferencia > 5% → estado `PENDING_REVIEW`, notificar supervisor
- Campo pendiente: `pedidos.precio_conflicto JSONB`

### Auto-assign de rutas (PENDIENTE de implementar)
- Vendor tiene `zona`, cliente tiene `zona`
- Cron 4AM: match automático por zona
- Supervisor puede reordenar manualmente vía dashboard

## Bull Queue — Plan de implementación (definido en sesión)

### Librería elegida: BullMQ (no Bull)
BullMQ es el sucesor activo de Bull. Usamos Redis 7 — compatible y recomendado.

### Una sola cola con prioridades
```
Cola: nexus:outbox
  P0 — ORDER_CREATED      priority: 10
  P0 — PAYMENT_RECORDED   priority: 10
  P1 — VISIT_CHECKIN      priority: 7
  P1 — VISIT_CLOSED       priority: 7
  P2 — PHOTO_UPLOADED     priority: 3  (no implementado en esta fase)
```

### Reglas hacia Odoo
- Rate limit: máx 2 jobs/segundo
- Concurrencia: máx 2 workers simultáneos
- Circuit breaker: 3 fallos → pausa 60s
- Backoff exponencial: 30s → 60s → 120s → 300s → 600s
- Max reintentos: 5 → Dead Letter Queue (status: DEAD en PostgreSQL)

### Flujo
```
Flutter → POST /sync/push → INSERT outbox (PENDING) → addJob BullMQ
BullMQ Worker → UPDATE outbox (SENDING) → Odoo JSON-RPC
  ✅ → UPDATE outbox (DONE) + pedidos.odoo_order_id
  ❌ → reencola con backoff (hasta 5x) → DEAD + alerta supervisor
```

### Validación de precio en worker
```
Por cada línea del pedido:
  diferencia < 5%  → enviar a Odoo
  diferencia > 5%  → PENDING_REVIEW, no enviar, notificar supervisor
```

### Estructura de archivos
```
queues/
  index.js              ← cola + conexión Redis
  worker.js             ← despacha por tipo + circuit breaker
  processors/
    order.js            ← ORDER_CREATED → sale.order Odoo
    payment.js          ← PAYMENT_RECORDED → account.payment Odoo
    visit.js            ← VISIT_CLOSED → field.visit Odoo
```

### Alcance de esta fase
- ✅ ORDER_CREATED
- ✅ PAYMENT_RECORDED
- ✅ VISIT_CLOSED
- ❌ PHOTO_UPLOADED (chunked upload es sistema separado)

## BullMQ — Implementación completada

### Archivos creados
```
queues/
  index.js              ← define cola "nexus-outbox" + prioridades + conexión Redis
  worker.js             ← worker principal + circuit breaker + dead letter
  processors/
    order.js            ← ORDER_CREATED → sale.order en Odoo + validación precios
    payment.js          ← PAYMENT_RECORDED → account.payment en Odoo
    visit.js            ← VISIT_CLOSED → field.visit en Odoo (graceful si módulo no existe)
```

### Fixes aplicados en implementación
- BullMQ no permite `:` en nombre de cola → renombrado a `nexus-outbox`
- Redis policy cambiada de `allkeys-lru` a `noeviction` → BullMQ no puede perder jobs por memoria
- Faltaban constraints UNIQUE en `pedidos.client_uuid` y `pagos.client_uuid` → agregados
- `pedidos.precio_conflicto JSONB` → columna agregada a la tabla

### Endpoints nuevos
- `POST /api/v1/sync/push` — recibe batch de eventos desde Flutter, idempotente por UUID
- `GET /api/v1/sync/status?uuids=...` — Flutter consulta estado de sus eventos

### Prueba exitosa
```
POST /sync/push → status: QUEUED ✅
Worker toma job → outbox.estado: SENDING ✅
Intenta Odoo → falla (cliente de prueba no existe en staging) → reintenta con backoff ✅
Mismo UUID enviado 2 veces → skipped: true ✅ (idempotencia funciona)
```

### Circuit breaker verificado en código
- 3 fallos consecutivos → pausa 60s → log de alerta
- Backoff exponencial: 30s → 60s → 120s → 300s → 600s
- 5 fallos → `outbox.estado = DEAD` → supervisor debe resolver
