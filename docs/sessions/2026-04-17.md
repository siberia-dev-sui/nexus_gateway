## Diagnóstico inicial — qué estaba fallando

Se analizaron los MDs de sesiones anteriores y el estado real del servidor. Se encontraron 3 problemas distintos:

### Problema 1 — Bug en `payment.js` (nuestro)
`payment.js:33` hacía `UPDATE pagos SET updated_at = NOW()` pero la tabla `pagos` **nunca tuvo columna `updated_at`** (solo `created_at`). Causa: schema.sql incompleto desde el inicio.

Afectó: 5 eventos PAYMENT_RECORDED → DEAD/SENDING con error `column "updated_at" of relation "pagos" does not exist`.

### Problema 2 — Lenn eliminó PAYMENT_RECORDED sin limpiar (commit 031c8d7)
Lenn eliminó el case `PAYMENT_RECORDED` del worker el 16/04 a las 16:06. Quedaron 4 eventos PAYMENT_RECORDED en SENDING — el worker ya no los procesa nunca. Nadie limpió la BD.

### Problema 3 — 7 eventos SENDING zombie por rebuild de Docker
Cuando el gateway se rebuildeó el 16/04, BullMQ perdió los jobs que estaban en vuelo. Los eventos quedaron en SENDING en PostgreSQL pero sin job en BullMQ — nunca iban a avanzar solos.

### Problema 4 — Odoo staging anterior caído/rate-limited
3 ORDER_CREATED en FAILED con error 503 (Odoo caído el 16/04 ~20:39) y 2 con 429 (rate limiting). Estos son del Odoo viejo (`19brandia.odoo.com`).

## Lo que se hizo hoy

### 1. Tarea 4 — Fix schema `pagos.updated_at`


Soluciona el bug que crasheaba `payment.js`. Los 5 eventos PAYMENT_RECORDED que murieron por este error ya no se pueden recuperar (Lenn además eliminó el processor), pero el schema queda correcto para futuros usos.

### 2. Tarea 1 — Tabla `gps_tracks`

**Commit:** `301fa18`

**Archivo:** `db/migrate_006_gps_tracks.sql`

```sql
CREATE TABLE IF NOT EXISTS gps_tracks (
  id          SERIAL PRIMARY KEY,
  vendedor_id INTEGER NOT NULL REFERENCES vendedores(id),
  lat         DOUBLE PRECISION NOT NULL,
  lng         DOUBLE PRECISION NOT NULL,
  accuracy    DOUBLE PRECISION DEFAULT 0,
  estado      TEXT DEFAULT 'en_ruta',
  captured_at TIMESTAMPTZ NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_gps_tracks_vendedor_date
  ON gps_tracks(vendedor_id, captured_at DESC);
```

Antes el GPS era **solo en Redis con TTL 2h** — al expirar se perdía todo. Ahora cada punto queda persistido en PostgreSQL para análisis histórico y el track del supervisor.

### 3. Tarea 5 — Limpieza de outbox zombies

**Commit:** `301fa18`

**Archivo:** `scripts/fix_zombies.js` (script de una sola ejecución)

**Resultado:**
- 4 PAYMENT_RECORDED SENDING/FAILED → **DEAD** (processor eliminado, nunca se procesarán)
- 3 eventos ORDER_CREATED/VISIT_CLOSED SENDING → **PENDING** + re-encolados en BullMQ

```
[FIX_ZOMBIES] PAYMENT_RECORDED → DEAD: 4
[FIX_ZOMBIES] SENDING → PENDING: 3
[FIX_ZOMBIES] Re-encolado: VISIT_CLOSED  660e8400-...
[FIX_ZOMBIES] Re-encolado: ORDER_CREATED e8778601-...
[FIX_ZOMBIES] Re-encolado: ORDER_CREATED 524a6786-...
```

**Por qué re-encolar en BullMQ y no solo cambiar el estado en BD:**
BullMQ y PostgreSQL son dos sistemas independientes. Si solo se cambia el estado en BD a PENDING, el worker nunca se entera — solo procesa lo que BullMQ le manda. Era necesario hacer `addToQueue()` para cada evento además del UPDATE.

### 4. Tarea 2 — `POST /api/v1/vendors/gps_track` + upgrade del PATCH

**Commit:** `301fa18`

**Nuevo endpoint — batch de puntos GPS (offline-first):**
```
POST /api/v1/vendors/gps_track   (JWT requerido)
Body: { points: [{ lat, lng, accuracy, estado, captured_at }] }
→ { status: "ok", saved: N }
```

Flutter acumula puntos GPS cuando no hay red y los manda en batch al recuperar señal. Todos se persisten en `gps_tracks`. El último punto del batch actualiza Redis `location:<vendedor_id>`.

**PATCH /vendors/location — upgrade silencioso:**
Sigue funcionando igual para Flutter (compatibilidad hacia atrás), pero ahora además inserta el punto en `gps_tracks`. Antes solo guardaba en Redis y se perdía al expirar.

**Helper `updateRedisLocation()`** extraído para que ambos endpoints compartan la misma lógica de Redis sin duplicar código.

### 5. Tarea 3 — `GET /api/v1/supervisor/vendor/:uuid/track`

**Commit:** `301fa18`

Endpoint para el mapa del supervisor — devuelve el recorrido completo de un vendedor en una fecha como GeoJSON LineString.

```
GET /api/v1/supervisor/vendor/:uuid/track?fecha=2026-04-17   (JWT supervisor/admin)
→ GeoJSON Feature {
    geometry: { type: "LineString", coordinates: [[-66.456, 10.123], ...] },
    properties: { vendedor_uuid, fecha, puntos: 48, distancia_km: 24.3 }
  }
→ Si no hay puntos: { geometry: null, properties: { puntos: 0 } }
```

Distancia calculada con **fórmula Haversine** entre puntos consecutivos ordenados por `captured_at ASC`.

### 6. Columna `rol` en `vendedores` — supervisores funcionales

**Commit:** `01247e0`

**Archivo:** `db/migrate_007_vendedores_rol.sql`

```sql
ALTER TABLE vendedores ADD COLUMN IF NOT EXISTS rol TEXT DEFAULT 'vendedor';
UPDATE vendedores SET rol = 'admin' WHERE email = 'dev@dev.com';
```

**El bug:** el login nunca leía la columna `rol`, así que `vendedor.rol` era siempre `undefined` → todos los JWTs decían `role: 'vendedor'` → los endpoints de supervisor siempre devolvían 403.

**El fix:** login y refresh token ahora incluyen `rol` en el SELECT y lo pasan al JWT.

```javascript
// login — antes
'SELECT id, uuid, nombre, email, password_hash, zona, activo FROM vendedores WHERE email = $1'

// login — ahora
'SELECT id, uuid, nombre, email, password_hash, zona, activo, rol FROM vendedores WHERE email = $1'
```

`dev@dev.com` queda con `rol = 'admin'` para poder probar los endpoints de supervisor. Los demás vendedores quedan con `rol = 'vendedor'` (valor por defecto).
