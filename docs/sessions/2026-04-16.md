### 2. POST /api/v1/visits — checkin y checkout

**Commit:** `cba8011`

Endpoint dedicado para el wizard de Flutter. Diferencia clave vs `sync/push`:
- Actualiza el estado de la `parada` (on_site / completed)
- Un solo evento, respuesta inmediata (no batch)
- Idempotencia independiente por operación

**Contrato:**
```
// CHECKIN
POST /api/v1/visits
{ tipo: "checkin", client_uuid, cliente_odoo_id, parada_id?, checkin_lat?, checkin_lng? }
→ { status: "ok", visita_uuid }

// CHECKOUT
POST /api/v1/visits
{ tipo: "checkout", client_uuid (nuevo UUID), visita_uuid (del checkin), notas? }
→ { status: "ok", visita_uuid }
```

**Fix en `visit.js`:** VISIT_CLOSED ahora usa `payload.visita_uuid` (del nuevo endpoint)
con fallback a `clientUuid` (compatibilidad con sync/push offline). Lee datos del
checkin desde PostgreSQL para completar el `field.visit` en Odoo.

**Tests verificados:**
- Sin campos → 400 ✅
- Checkin → visita creada, parada en `on_site` ✅
- Mismo UUID → `skipped: true` (idempotencia) ✅
- Checkout → visita cerrada, parada en `completed`, outbox VISIT_CLOSED encolado ✅

### 3. POST /api/v1/auth/refresh — sesión continua

**Commit:** `2db37fb`

Sin este endpoint, el JWT de 24h expiraba y el vendedor quedaba sin sesión en campo
(Venezuela, señal intermitente). Ahora el flujo es:

```
Login → access JWT (24h) + refresh token (30 días)
Cuando JWT expira → POST /auth/refresh → nuevo JWT + nuevo refresh token (rotación)
```

**Implementación:**
- Refresh token = `crypto.randomBytes(40).toString('hex')` (80 chars)
- Se guarda SHA-256 del token en tabla `refresh_tokens` (nunca el raw)
- Rotación: cada refresh revoca el token anterior e issue uno nuevo
- Vendedor inactivo → 401

**Login ahora devuelve:**
```json
{
  "token": "<JWT 24h>",
  "refresh_token": "<token 30 días>",
  "expires_in": 86400,
  "vendedor": { ... }
}
```

**Tests verificados:** login emite ambos tokens ✅, refresh devuelve par nuevo ✅,
token original revocado ✅, token basura → 401 ✅.

### 6. POST /api/v1/photos/upload — upload chunked de fotos

**Commit:** `d560d53` (primera versión — tenía odooCall, ver sección 7)

Nuevo endpoint para subir evidencia fotográfica desde Flutter. No requiere
dependencia externa (`@fastify/multipart`): usa JSON + base64, que es más
compatible con el patrón offline-first.

**Protocolo (un request por chunk):**
```
POST /api/v1/photos/upload  (JWT requerido)
{
  visita_uuid,   // nexus_uuid de la visita
  foto_uuid,     // UUID fijo para todos los chunks de esta foto
  filename,      // "foto_001.jpg"
  chunk_index,   // 0-based
  total_chunks,
  data           // base64 del chunk
}
```

- Primer chunk → INSERT en tabla `fotos` (tracking de chunks)
- Chunks siguientes → `UPDATE fotos SET chunks_recibidos + 1`
- Último chunk → ensambla todos los chunks en `/tmp/nexus_photos/<visita_uuid>/`,
  sube a Odoo vía módulo, marca `fotos.upload_completo = true`
- Respuesta parcial: `{ status: "partial", foto_uuid, chunks_recibidos }`
- Respuesta final:   `{ status: "complete", foto_uuid, odoo_attachment_id }`
- Idempotente: si el chunk ya existe en disco, no lo reescribe

**Tabla `fotos` (ya existía en el schema):** uuid, visita_id, vendedor_id,
filename, storage_path, size_bytes, upload_completo, chunks_total,
chunks_recibidos.

### 7. Refactor crítico — todos los workers pasan por el módulo nexus_mobile

**Commit:** `ad83984`

**El problema detectado:** el Claude del servidor implementó `order.js` usando
`odooCall` directo al ORM de Odoo (lo mismo que ya estaba en `payment.js` y
`visit.js`). Esto viola la regla arquitectónica fundamental:

```
TODO pasa por: Gateway → módulo nexus_mobile → Odoo
NUNCA:         Gateway → Odoo (ORM directo)
```

**Por qué importa:** si Lenn cambia un campo o modelo en Odoo, el gateway se
rompe. Con el módulo como intermediario, Lenn absorbe los cambios internamente
y el contrato del gateway (los endpoints del módulo) permanece estable.

**Lenn ya tenía los 5 endpoints del módulo listos:**

| Endpoint del módulo | Reemplaza |
|---------------------|-----------|
| `POST /nexus/api/v1/create_order` | `odooCall('sale.order', 'create', ...)` + `action_confirm` + `read` |
| `POST /nexus/api/v1/create_visit` | `odooCall('field.visit', 'create', ...)` |
| `POST /nexus/api/v1/create_payment` | `odooCall('account.payment', 'create', ...)` + `action_post` + journal lookup |
| `POST /nexus/api/v1/upload_attachment` | `odooCall('ir.attachment', 'create', ...)` |
| `POST /nexus/api/v1/all_vendor_clients` | ya migrado en sesión anterior |

**Contratos relevantes:**
```javascript
// Pedido — módulo confirma internamente, devuelve name directamente
odooPost('/nexus/api/v1/create_order', {
  client_uuid, cliente_odoo_id, visita_uuid, vendor_nexus_uuid,
  lines: [{ product_id, qty, price_unit, name }]
})
→ { order_id, name, created }

// Visita — idempotente por nexus_uuid, tipo controla si abre o cierra
odooPost('/nexus/api/v1/create_visit', {
  nexus_uuid, tipo,   // 'checkin' | 'VISIT_CLOSED'
  cliente_odoo_id, vendor_nexus_uuid,
  checkin_at, checkout_at, checkin_lat, checkin_lng, notas
})
→ { visit_id, created }

// Pago — módulo maneja journal + action_post
odooPost('/nexus/api/v1/create_payment', {
  client_uuid, cliente_odoo_id, visita_uuid, vendor_nexus_uuid, monto, metodo
})
→ { payment_id, created }

// Foto — gateway ensambla chunks, llama esto con el base64 completo
odooPost('/nexus/api/v1/upload_attachment', {
  visita_uuid, nombre, datos_b64, mimetype
})
→ { attachment_id, created }
```

**Mecanismo de inyección (`worker.js`):**

El worker recibe `odooPost` igual que ya recibía `odooCall`, pero ahora
los processors lo usan para todo:

```javascript
// worker.js
let globalOdooPost = null
function setOdooPost(fn) { globalOdooPost = fn }

// En el switch:
result = await processOrder(job, globalOdooPost)   // antes: globalOdooCall

// server.js al arrancar:
setOdooCall(odooCall)   // legacy — mantenido
setOdooPost(odooPost)   // activo — todos los processors lo usan
```

**`vendor_nexus_uuid` en todos los processors:**
No se lee del JWT directamente (el job de BullMQ no lleva el request original).
Se resuelve con una query PostgreSQL usando el outbox:
```javascript
SELECT v.uuid AS vendor_uuid
FROM outbox o JOIN vendedores v ON v.id = o.vendedor_id
WHERE o.client_uuid = $1
```

**Estado actual de ORM directo (deuda pendiente):**
- `crons/sync_vendors.js` → `odooCall('nexus.vendor', ...)` ⚠️
- `crons/sync_prices.js` → `odooCall('product.product', ...)` ⚠️

Los workers (orders, payments, visits, photos) están 100% limpios.

## Estado arquitectónico al cierre — ORM directo vs módulo

### La regla
**Todo pasa por el módulo `nexus_mobile` de Lenn. Nunca `odooCall` directo al ORM.**

```
Gateway → módulo nexus_mobile → Odoo   ✅
Gateway → Odoo (ORM directo)           ❌
```

Si Lenn cambia un campo o modelo en Odoo, el módulo absorbe el cambio internamente.
El gateway solo conoce el contrato de los endpoints del módulo — que es estable.
