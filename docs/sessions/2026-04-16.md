### 2. POST /api/v1/visits — checkin y checkout

**Commit:** `cba8011`

Endpoint dedicado para el wizard de Flutter. Diferencia clave vs `sync/push`:
- Actualiza el estado de la `parada` (on_site / completed)
- Un solo evento, respuesta inmediata (no batch)
- Idempotencia independiente por operación

**Contrato:**
```
// CHECKIN
POST /api/v1/visits
{ tipo: "checkin", client_uuid, cliente_odoo_id, parada_id?, checkin_lat?, checkin_lng? }
→ { status: "ok", visita_uuid }

// CHECKOUT
POST /api/v1/visits
{ tipo: "checkout", client_uuid (nuevo UUID), visita_uuid (del checkin), notas? }
→ { status: "ok", visita_uuid }
```

**Fix en `visit.js`:** VISIT_CLOSED ahora usa `payload.visita_uuid` (del nuevo endpoint)
con fallback a `clientUuid` (compatibilidad con sync/push offline). Lee datos del
checkin desde PostgreSQL para completar el `field.visit` en Odoo.

**Tests verificados:**
- Sin campos → 400 ✅
- Checkin → visita creada, parada en `on_site` ✅
- Mismo UUID → `skipped: true` (idempotencia) ✅
- Checkout → visita cerrada, parada en `completed`, outbox VISIT_CLOSED encolado ✅

### 3. POST /api/v1/auth/refresh — sesión continua

**Commit:** `2db37fb`

Sin este endpoint, el JWT de 24h expiraba y el vendedor quedaba sin sesión en campo
(Venezuela, señal intermitente). Ahora el flujo es:

```
Login → access JWT (24h) + refresh token (30 días)
Cuando JWT expira → POST /auth/refresh → nuevo JWT + nuevo refresh token (rotación)
```

**Implementación:**
- Refresh token = `crypto.randomBytes(40).toString('hex')` (80 chars)
- Se guarda SHA-256 del token en tabla `refresh_tokens` (nunca el raw)
- Rotación: cada refresh revoca el token anterior e issue uno nuevo
- Vendedor inactivo → 401

**Login ahora devuelve:**
```json
{
  "token": "<JWT 24h>",
  "refresh_token": "<token 30 días>",
  "expires_in": 86400,
  "vendedor": { ... }
}
```

**Tests verificados:** login emite ambos tokens ✅, refresh devuelve par nuevo ✅,
token original revocado ✅, token basura → 401 ✅.

## Problema arquitectónico identificado — sync_clients usa ORM directo

### El contexto

La arquitectura del proyecto establece que **todo debe pasar por el módulo `nexus_mobile`
de Lenn**, no llamar a Odoo directamente. El módulo es la capa de abstracción:

```
Gateway → módulo nexus_mobile → Odoo
```

### Lo que está bien (pasa por el módulo)
- Catálogo → `odooPost('/nexus/api/v1/catalog')`
- Sync manual de clientes → `odooPost('/nexus/api/v1/vendor_clients')`
- Health check → `odooPost('/nexus/api/v1/health')`

### Lo que está mal (llama al ORM directo)
- `crons/sync_clients.js` → `odooCall('nexus.vendor', ...)` + `odooCall('res.partner', ...)`
- `crons/sync_vendors.js` → `odooCall('nexus.vendor', ...)`
- `crons/sync_prices.js` → `odooCall('product.product', ...)`
- Workers BullMQ → `odooCall('sale.order', ...)`, `odooCall('account.payment', ...)`, etc.

### Por qué es un problema
Si Lenn cambia un campo en Odoo o en el módulo, hay que tocar el gateway también.
Con el módulo como intermediario, Lenn absorbe los cambios sin afectar el gateway.

### La situación con sync_clients específicamente

El módulo solo expone **un endpoint por vendedor**:
```
POST /nexus/api/v1/vendor_clients  { nexus_uuid: "..." }
→ clientes de UN vendedor
```

No existe endpoint batch. Para migrar el cron hay dos opciones:

**Opción A — Lenn agrega endpoint batch (recomendada)**
```
POST /nexus/api/v1/all_vendor_clients
→ { vendors: [{ nexus_uuid, clients: [...] }] }
```
Una sola llamada, el cron lo usa. Limpio y eficiente.

**Opción B — El cron hace loop con el endpoint existente (disponible hoy)**
```javascript
const vendors = await db.query('SELECT nexus_uuid FROM vendedores WHERE activo = true')
for (const v of vendors.rows) {
  const data = await odooPost('/nexus/api/v1/vendor_clients', { nexus_uuid: v.nexus_uuid })
  // upsert clientes...
}
```
Funciona pero son N llamadas a Odoo (una por vendedor). Con 120 vendedores = 120 requests.

**Decisión pendiente:** coordinar con Lenn si agrega el endpoint batch (Opción A)
o implementamos el loop como solución temporal (Opción B).
